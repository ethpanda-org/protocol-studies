# EIP-4844

## 概述

L2 Rollup 必须允许无需许可地重建 L2 状态。为了实现这一目标，他们必须确保数据可用性，它指的是已改变状态的交易输入的全局排序。
在 4844 在主网上运行之前，Rollup 仅通过交易中称为 `calldata` 的部分存储数据。

[EIP-4844](https://eips.ethereum.org/EIPS/eip-4844) 引入了“blob”，它修改了 [execution-specs](https://github.com/ethereum/execution-specs/blob/0f9e4345b60d36c23fffaa69f70cf9cdb975f4ba/src/ethereum/london/fork_types.py#L49) 中定义的当前交易和 区块结构，分别为[共识规范](https://github.com/ethereum/consensus-specs/blob/1a33bf8a00bb593d61a9128c929b8dcacef0b437/specs/deneb/beacon-chain.md#beaconblockbody)。

EIP-4844 引入了一种新的交易类型，即 blob 携带交易。

引用自 [proto-danksharding-faq(aka EIP-4844)](https://notes.ethereum.org/@vbuterin/proto_danksharding_faq#What-is-proto-danksharding-aka-EIP-4844)：

> proto-danksharding 引入的主要功能是新的交易类型，我们将其称为 blob 携带交易。携带交易的 blob 与常规交易类似，只不过它还携带一条称为 blob 的额外数据。 blob 非常大(~125 kB)，并且比类似数量的呼叫数据便宜得多。然而，blob 数据无法被 EVM 执行访问； EVM 只能查看承诺到blob。

本页概述了EIP，提供代码参考以了解此硬分叉带来的变化。它针对的是有抱负的以太坊核心开发人员，有兴趣理解以太坊更新后的代码。

## 核心变化

### 新 gas 处理

`calldata` 是扩展以太坊的主要瓶颈，因为在 `calldata` 中存储数据成本高昂，并且计划使其[甚至更昂贵](https://eips.ethereum.org/EIPS/eip-7623)。

[EIP-4844](https://eips.ethereum.org/EIPS/eip-4844) 引入了“blob”一种在交易中存储数据的新方法，该方法比使用 `calldata` 便宜得多。这是因为blob将使用一种新型的gas，独立于传统的gas，并遵循自己的定价规则。

EIP-4844 标志着朝着[实现更高效的 gas 使用](https://ethresear.ch/t/on-increasing-the-block-gas-limit/18567) 迈出了第一步。

这些 EIP 的目标是实现更小、变量更少的区块大小。这个目标有两个主要原因：它为单独的质押者提供保护，并为未来的 blob 存储创造额外的容量。

但是我们怎样才能实现更小、变量更少的区块大小呢？一种方法是增加非零 `calldata` 字节的成本，如 [EIP-7623](https://eips.ethereum.org/EIPS/eip-7623) 中提出的。提高 `calldata` 成本会减少最大可能的区块大小(假设 `gas limit` 保持在 30,000,000，并且每个 `calldata` 字节的 gas 成本增加，导致区块内容纳的字节数减少)。此外，增加的`calldata`成本进一步鼓励使用blob作为数据可用性。

探索新的、有效的方法来管理 gas 和区块 gas 限制仍然是[积极讨论的主题](https://ethresear.ch/t/on-increasing-the-block-gas-limit/18567)。

### 共识规范的变化

将 EIP-4844 引入共识层的升级已标记为 **Deneb**。

#### 1. Beacon 区块数据结构变更

`BeaconBlockBody`(表示共识层中 交易的 区块的数据结构)已更新，包含 2 个新字段：`blob_kzg_commitments` 和 `execution_payload` 字段的更新版本。

请参阅[此处](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/beacon-chain.md#beaconblockbody)。

需要注意的是，blob 不会被添加到它们所属的区块中。相反，它们将通过使用 `blob_kzg_commitments` 在区块中引用。 blob 通过“sidecar”单独传播到网络。

KZG 承诺是一种加密承诺类型，因其在创建和验证数据可用性证明和正确性方面的效率而特别有用。

KZG 承诺能够证明特定数据片段包含在集合中，而无需透露整个数据集。这对于可扩展性解决方案特别有用，因为它不需要每个节点存储整个区块链来证明交易的正确性。

检查以下链接以了解更多信息：

- [KZG 仪式](https://github.com/ethereum/kzg-ceremony)
- [基于算术哈希的 KZG 替代方案，用于原型 danksharding (EIP-4844)](https://ethresear.ch/t/arithmetic-hash-based-alternatives-to-kzg-for-proto-danksharding-eip-4844/13863)
- [分片多方计算仪式](https://ethresear.ch/t/sharding-multi-party-computation-ceremony/11779)

#### 2. blob sidecar

如 [EIP-4844](https://eips.ethereum.org/EIPS/eip-4844#consensus-layer-validation) 中所述，blob 被引用，但未在 [信标区块主体](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/beacon-chain.md#beaconblockbody) 内完全编码。 blob 不是将完整内容嵌入到正文中，而是单独传播，如“sidecar”。

sidecar数据结构请参见[此处](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/p2p-interface.md#blobsidecar)

在将 blob sidecar 传播到 p2p 网络之前，节点必须验证 blob 是否有效(请参阅[此处](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/p2p-interface.md#blob-subnets))。

当节点向网络请求 blob 时，网络会以 `BlobSidecar` 数据结构进行响应。
接收 sidecar 的节点应验证 blob sidecar 格式正确，具有有效的包含证明，并且正确。在读取 blob 中的数据之前预期的 KZG 承诺。

这种“sidecar”设计为通过黑盒[is_data_available](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/fork-choice.md#is_data_available)进一步增加数据提供了前向兼容性，这意味着在未来的升级中`is_data_available()`实现可能会改变，例如包括数据可用性采样(DAS)，并且与此功能交互的系统是不需要了解函数的内部实现(在编程术语中，`is_data_available()`可以被视为一个接口方法，其实现可能会随着时间的推移而改变)。

#### 3. 包含 KZG 承诺版本哈希

当新的区块载荷提交到链时，共识层(CL，也称为信标链)调用 [`process_execution_payload`](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/beacon-chain.md#modified-process_execution_payload) 函数。该函数负责对区块的载荷执行一些有效性检查，然后通过`verify_and_notify_new_payload`函数调用执行层(EL)。

一旦调用，EL 将：

- 验证区块载荷
- 在区块载荷内执行交易
- 更新state，这是执行交易的结果

对于 EIP-4844，`process_execution_payload` 添加要传递给 `verify_and_notify_new_payload` 函数的参数 `versioned_hashes`。

`versioned_hashes` 是 KZG 承诺的每个 blob 的 [哈希](https://github.com/ethereum/consensus-specs/blob/dev/specs/deneb/beacon-chain.md#modified-process_execution_payload) 数组。

#### 4. 执行载荷和 执行载荷标头

执行载荷 [这是从共识层发送到执行层上的状态转换函数的 载荷](https://github.com/ethereum/consensus-specs/blob/1a33bf8a00bb593d61a9128c929b8dcacef0b437/specs/deneb/beacon-chain.md#execution-payload) 扩展了两个新的 64 位无符号整数字段 `blob_gas_used` 和`excess_blob_gas`。 blob 交易有自己的 gas 定位规则(如上一节所述)，除了执行所花费的正常 gas 费用之外，节点还必须考虑在 blob 中存储数据所花费的 gas 交易(请参阅下面的**执行规范的更改**)。

- 参见[1](https://github.com/ethereum/consensus-specs/blob/1a33bf8a00bb593d61a9128c929b8dcacef0b437/specs/deneb/beacon-chain.md#executionpayload)和[2](https://github.com/ethereum/consensus-specs/blob/1a33bf8a00bb593d61a9128c929b8dcacef0b437/specs/deneb/beacon-chain.md?plain=1#L149)

#### 5.新增区块标头检查

请参阅 go-ethereum 文件“consensus.go”以了解添加到区块标头的检查：

- [链接](https://github.com/ethereum/go-ethereum/blob/0a2f33946b95989e8ce36e72a88138adceab6a23/consensus/beacon/consensus.go#L274C1-L284C3)

### 执行规范的变化

将 EIP-4844 引入执行层的升级已标记为**坎昆**。

#### 1.在状态转换函数里面添加检查

EL 现在必须检查 blob 特定字段对于将在状态转换函数 (STF) 中执行的每个交易是否有效。

检查包括(有关规范代码，请参阅[此处](https://eips.ethereum.org/EIPS/eip-4844#execution-layer-validation))：

- 检查签名者是否有足够的余额来支付交易 gas 费用和 blob gas 费用

```python
# modify the check for sufficient balance
max_total_fee = tx.gas * tx.max_fee_per_gas
if get_tx_type(tx) == BLOB_TX_TYPE:
    max_total_fee += get_total_blob_gas(tx) * tx.max_fee_per_blob_gas
assert signer(tx).balance >= max_total_fee
```

- 检查 blob 交易是否包含至少 1 个有效的 `blob_versioned_hash`(请参阅 CL 更改)并且它们的格式是否正确

```python
assert len(tx.blob_versioned_hashes) > 0
```

- 检查用户是否愿意至少支付当前的 blob 基本费用

```python
assert tx.max_fee_per_blob_gas >= get_blob_base_fee(block.header)
```

最后，EL STF 必须跟踪用于 blob 交易的 gas(与 EIP1559 交易已经发生的情况相同)。

```python
blob_gas_used += get_total_blob_gas(tx)
```

#### 2.新增交易数据结构(BLOB_TX_TYPE)

在 EIP-4844 之前，交易数据结构如下([以太坊执行规范仓库](https://github.com/ethereum/execution-specs/blob/0f9e4345b60d36c23fffaa69f70cf9cdb975f4ba/src/ethereum/london/fork_types.py#L49C1-L108C2))：

```python
@slotted_freezable
@dataclass
class LegacyTransaction:
  nonce: U256
  gas_price: Uint
  gas: Uint
  to: Union[Bytes0, Address]
  value: U256
  data: Bytes
  v: U256
  r: U256
  s: U256

@slotted_freezable
@dataclass
class AccessListTransaction:
  chain_id: U64
  nonce: U256
  gas_price: Uint
  gas: Uint
  to: Union[Bytes0, Address]
  value: U256
  data: Bytes
  access_list: Tuple[Tuple[Address, Tuple[Bytes32, ...]], ...]
  y_parity: U256
  r: U256
  s: U256

@slotted_freezable
@dataclass
class FeeMarketTransaction:
  chain_id: U64
  nonce: U256
  max_priority_fee_per_gas: Uint
  max_fee_per_gas: Uint
  gas: Uint
  to: Union[Bytes0, Address]
  value: U256
  data: Bytes
  access_list: Tuple[Tuple[Address, Tuple[Bytes32, ...]], ...]
  y_parity: U256
  r: U256
  s: U256

Transaction = Union[
  LegacyTransaction, AccessListTransaction, FeeMarketTransaction
]
```

`calldata` 编码在交易类的 `data` 字段内。 `calldata`可由EVM访问，其编码和解码格式在[合约ABI](https://docs.soliditylang.org/en/latest/abi-spec.html)中指定。

EIP-4844会在交易类中引入另外两个字段，分别是`max_fee_per_blob_gas`和`blob_versioned_hashes`。
EIP-4844 引入了一个新的交易类型，其中 `TransactionType == BLOB_TX_TYPE` 和 `TransactionPayload` 是下面类的 rlp 编码。

对于 EIP-4844，新的交易数据结构将如下：

```python
@slotted_freezable
@dataclass
class BlobTransaction: # the class name may change
  chain_id: U64
  nonce: U256
  max_priority_fee_per_gas: Uint
  max_fee_per_gas: Uint
  gas: Uint
  to: Union[Bytes0, Address]
  value: U256
  data: Bytes
  access_list: Tuple[Tuple[Address, Tuple[Bytes32, ...]], ...]
  max_fee_per_blob_gas: U256
  blob_versioned_hashes: VersionedHash
  y_parity: U256
  r: U256
  s: U256
```

## 关于之前为提高 Rollup 的可扩展性而进行的尝试/研究的说明

为了增强可扩展性，提出了两个以太坊改进建议(EIP)：[EIP-4488](https://eips.ethereum.org/EIPS/eip-4488)和[EIP-4490](https://github.com/ethereum/EIPs/pull/4490)。

两个 EIP 的核心目标都是降低与交易 `calldata` 相关的 gas 成本。 EIP-4490 建议将 `calldata` 成本从每字节 16 个 gas 降低到 6 个。此更改虽然易于实施，但可能会将最大区块大小增加到 5MB。相反，EIP-4488 提出了更显着的减少，从每字节 16 个 gas 减少到 3 个 gas，但对每个区块中允许的 `CALLDATA` 数量引入了上限，以有效限制区块的大小。

有关这两项提案的更详细讨论，请考虑查看 Tim Beiko 的 [这条推文](https://twitter.com/TimBeiko/status/1464268807854190594)。

请注意，上述 EIP [EIP-4488](https://eips.ethereum.org/EIPS/eip-4488) 和 [EIP-4490](https://github.com/ethereum/EIPs/pull/4490) 现已弃用，取而代之的是引入 blob。
